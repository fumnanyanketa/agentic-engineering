🎯 Objective: Build a real evaluation practice: a repeatable, written-down way to measure whether your agent does its job. Many leading practitioners call this the single most important new skill in the field.

One word to define first. An eval (short for evaluation) is a test that measures how well your AI system does its job. Instead of poking the system by hand and saying "looks good," you collect a set of example tasks, run your system on them, and score the results. That score is a number you can track over time, so you can tell whether a change made things better or worse. Without evals, "it works" is just a feeling.

Why evaluation is so hard (and so valuable)

In ordinary software, a test has one right answer: 2 plus 2 should equal 4. Language models are different. For an open-ended task like "summarize this report," there are many valid answers and no single correct one. That is what makes evaluation "a much bigger problem" here than in classic machine learning. Because it is hard, most teams skip it, and that is exactly why doing it well sets you apart. (a survey of the AI engineering stack)

Start with error analysis, not with imagined tests

The first instinct is to sit down and dream up tests. Resist it. The most useful starting activity is error analysis, which simply means: run your system on real tasks, then read the outputs carefully and write down every failure you actually see. The machine learning engineer Hamel Husain calls error analysis "the most important activity in evals" and the place where "you build your moat" (your durable advantage). The rule that follows is: write tests for the errors you discover in real transcripts, not the errors you imagine. Imagined failures waste effort on problems you do not have. (Husain and Shankar, an FAQ on AI evals)

A transcript (sometimes called a trace) is the full record of what your agent did on one task: what it was asked, what it thought, which tools it called, and what it answered. Reading transcripts is the work. There is no shortcut.

The main evaluation methods

Once you know your real failure modes, you can choose how to grade. Common methods, from most objective to most subjective:

  • Ground-truth or benchmark grading. You have a known correct answer (or a clear automatic check, like "did the code run?") and compare against it. The most reliable when it is possible.
  • LLM-as-judge. You use a second language model to score the first model's output against instructions. Powerful and scalable, but it must be checked carefully (more on this below).
  • Human evaluation. A person reads and scores. Slow and expensive, but the gold standard you measure the others against.
  • Pairwise comparison. Instead of scoring one answer in isolation, you show two answers and ask which is better. The engineer Eugene Yan notes that relative judgments ("A is better than B") tend to be more reliable than absolute scores, because they are easier for both people and models to make consistently. (Eugene Yan on LLM-evaluators)

Prefer binary labels over 1-to-5 scales

When you label outputs, prefer a simple pass or fail over a 1-to-5 rating (a so-called Likert scale, a numeric agreement scale). The reason is that the difference between a 3 and a 4 is subjective and inconsistent, even for the same person on different days, so the numbers drift and become meaningless. Pass or fail forces a clear decision and is faster to make. Husain also recommends naming one domain expert as the single "benevolent dictator" who owns labeling, so your standard stays consistent instead of varying across many labelers. (Husain on binary versus Likert)

Grading agents: score the outcome, not the exact path

Agents add a twist. An agent can reach a correct answer by many different routes (different tool calls in a different order). It is tempting to check that it followed one exact sequence of tools, but that is "too rigid and results in overly brittle tests" that fail whenever the agent finds a different valid path. The durable rule: use the trajectory (the path of tool calls) for debugging when something breaks, but grade task completion, meaning did it actually accomplish the goal. (a guide to demystifying evals for AI agents · trajectory evaluations)

LLM-as-judge: use it, but do not trust it blindly

Using a model to grade other models scales beautifully, but model judges have known, measurable biases. Reported examples include:

  • Verbosity bias: a tendency to prefer longer answers, well over 90 percent of the time in some tests, even when the longer answer is worse.
  • Position bias: a tendency (around 70 percent in some tests) to prefer whichever answer was shown first.
  • Self-enhancement bias: a tendency to rate a model's own style of output higher, on the order of 10 percent.

Despite this, judges can be good: on some public datasets a strong model judge matched human-to-human agreement (for example around 85 percent agreement on one benchmark, slightly above the 81 percent that humans reached with each other), while on other datasets it missed many real defects. The takeaway is not "never use a judge," it is calibrate the judge against human labels before you trust it: have your expert label a sample by hand, then check that the judge agrees with that human before you rely on it at scale. (Eugene Yan on LLM-evaluators)

Automated evals are necessary but not sufficient

Even a great automated suite has a blind spot. As Simon Willison puts it, "just because code passes tests doesn't mean it works as intended," and "the one thing you absolutely cannot outsource to the machine is testing that the code actually works." Pair your automated evals with manual testing: actually drive the running system and watch it work with your own eyes. The fuller discipline of this is in the coding-agents module. (Willison on designing agentic loops)

Start small

You do not need thousands of cases to begin. Start with roughly 20 to 50 tasks drawn from real failures you observed, each paired with a reference solution that proves the task is actually solvable. Then read the transcripts: you cannot trust a grader you have never inspected. (demystifying evals for AI agents)

Lab 12: Take the agent system you built in Phase 4 and: (1) collect real transcripts by running it on real tasks, (2) do error analysis and group the failures you actually see into a short list of categories, (3) build an eval set of 30 to 50 cases, each with a reference outcome, (4) write graders (a mix of simple code-based checks and an LLM-judge that you have calibrated against your own human labels), and (5) run it and report a single pass rate. That eval set becomes your capstone's report card.

✅ Best practices:

  • Build evals from real, observed failures, not imagined ones.
  • Use binary pass or fail labels rather than 1-to-5 scales.
  • Appoint one expert as the single labeling authority.
  • Grade the outcome (did it complete the task), not the exact tool path.
  • Read the transcripts before trusting any grader.

⛔ Pitfalls:

  • Shipping with no evals at all, so "it works" is just a vibe.
  • Using fuzzy 1-to-5 scores that drift and mean nothing.
  • Trusting an LLM-judge you never calibrated against humans.
  • Brittle checks that demand one exact sequence of tool calls.
  • Outsourcing all labeling, which breaks the feedback loop you need.

⚖ Debate to know: Some advocate eval-driven development (write all your evals before you write the code). Others, including Husain, argue this is counterproductive because the set of possible failures is effectively infinite, so you should start from the failures you actually observe (error-analysis-first). The practitioner-grounded position leans error-analysis-first, so treat "write all your evals upfront" with healthy suspicion.

🏁 Milestone: A real eval suite with a reported pass rate, built from observed failures, using a judge you have calibrated against human labels.
