🎯 Objective: Understand and use the Model Context Protocol (MCP), the open, vendor-neutral standard for connecting agents to tools, data, and prompts, and ship a small server of your own.

By now you can give an LLM (Large Language Model, the kind of AI that powers chat assistants) tools, retrieval, and memory. The next problem is practical: every time you want to connect a model to a new tool or data source, you have to write custom plumbing. MCP fixes that by defining one shared way to make those connections, so the work is reusable across tools and across platforms.

What MCP is

MCP stands for Model Context Protocol. A protocol is simply an agreed set of rules for how two programs talk to each other (much like a shared language). MCP is an open standard for secure, two-way connections between AI applications and the tools and data they need. A common analogy is that MCP is "the USB-C port for AI applications": just as USB-C lets any compatible device plug into any compatible port, MCP lets any compatible AI app plug into any compatible tool or data source. (MCP docs)

MCP was first announced in November 2024, but it is no longer one company's project. It has been adopted across the industry, and in December 2025 it was donated to the Agentic AI Foundation, a vendor-neutral body under the Linux Foundation, so it is now governed as a shared community standard. Treat it the way you would treat any open web standard: a common protocol that many vendors implement, not a single product. (Linux Foundation: forming the Agentic AI Foundation)

Why it matters: the N times M problem

Imagine you have N different AI applications and M different tools or data sources you want them to use. Without a shared standard, you might have to build a custom connector for every pairing, which is N times M separate integrations. That number explodes quickly: 10 apps and 10 tools would mean 100 connectors to build and maintain.

MCP turns that into an N plus M problem instead. Each application learns to speak MCP once, and each tool is wrapped as an MCP server once. After that, any MCP application can use any MCP server with no extra glue code. Build one server, and every MCP client can use it. That is the core payoff of the standard.

How MCP is put together

MCP follows a client-server design, a common pattern where one program (the client) requests things and another (the server) provides them. The pieces are:

  • A host is the AI application you actually use. Neutral examples include desktop chat apps, code editors, and integrated development environments (an IDE is the software a programmer writes code in).
  • Inside the host runs an MCP client, which holds an open connection to a server.
  • An MCP server is a small program that exposes capabilities (such as access to your files or a database) in the MCP format.

The two sides talk using JSON-RPC 2.0. That is just a standard, lightweight format for sending requests and getting replies, where the messages are written in JSON (a simple, human-readable way to structure data). You do not need to memorize the format; you only need to know that it is the agreed wire format both sides use.

The three server primitives

An MCP server can offer three kinds of capability, often called its primitives (the basic building blocks):

  • Tools: functions the model can call to take an action, such as "send an email" or "run a query." These are the executable abilities.
  • Resources: structured data the model can read into its context, such as a file's contents or a record from a database.
  • Prompts: reusable prompt templates or canned workflows that a user or app can pull in on demand.

The protocol also defines some capabilities on the client side, but when people say "the three primitives," they mean these three server features. (Wikipedia: Model Context Protocol)

Putting MCP to work, vendor-neutrally

Because MCP is a shared standard, your servers and clients work across many platforms rather than locking you to one. A practical first project is to connect your agent to an existing server (common examples wrap a filesystem, a code-hosting service, or a database), and then to write your own server that wraps the retrieval tool you built in Module 5. Most modern agent platforms and coding agents now speak MCP, so a server you write once can be reused widely. (a year of MCP: a review of adoption)

Lab 9: Two parts. First, connect your agent to two pre-built MCP servers (pick any two, such as a filesystem server and a database server) and call them from your agent. Second, write a minimal MCP server of your own that exposes one tool and one resource, then call it from your agent. By the end you will understand both sides of the protocol: using servers and building them.

✅ Best practices:

  • Prefer MCP whenever an integration is reusable, so you build it once instead of repeatedly.
  • Give every server tool a clear description and a sensible name, applying the same naming rules from Module 4.
  • Treat tool descriptions from any server as untrusted input, because that text enters your model's context. See Module 14 on tool poisoning.
  • Lean on the shared standard: write servers that any compliant client can use, rather than tying them to one app.

⛔ Pitfalls:

  • Rebuilding bespoke, one-off integrations for things MCP already standardizes.
  • Trusting third-party MCP servers blindly. Their tool descriptions flow straight into your model's context, so a malicious one can try to manipulate your agent.
  • Assuming MCP belongs to a single vendor. It is an open standard stewarded by a foundation, so build to the spec, not to one product.

🏁 Milestone: Your agent uses external capabilities through MCP, and you have shipped a working MCP server that another MCP client could reuse.
