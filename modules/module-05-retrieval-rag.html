A model only knows what it learned during training. Retrieval lets you feed it the right facts at the right moment, your private documents or fresh information, without retraining anything.

🎯 Objective: Learn to ground a model in information it was never trained on (your private documents, fresh data) by fetching the right facts at the right moment and placing them into the prompt. You will build a retrieval pipeline, measure how good it is, and improve it with techniques that work no matter which model or database you use.

A model finishes training on a fixed snapshot of text and learns nothing after that. It does not know your company's internal wiki, your customer records, or anything that happened after its training cutoff (the date its training data ends). RAG stands for Retrieval-Augmented Generation, which means: before the model answers, you retrieve relevant pieces of text from your own sources and paste them into the prompt, so the model can generate its answer using those pieces. No retraining required. When a new document appears, it is usable the moment you add it to your store.

RAG is the standard answer to "the model needs to know my stuff." It is cheaper, faster, and more current than retraining, and it lets you cite exactly where an answer came from.

The moving parts

A basic RAG pipeline has a small number of pieces. Here is each one in plain terms.

  • Chunking. You cannot store whole books as single units, so you split your documents into smaller passages called chunks, perhaps a few paragraphs each. How you cut them matters: a chunk should be big enough to make sense on its own but small enough to be specific.
  • Embeddings. An embedding is a list of numbers that captures the meaning of a piece of text. Texts with similar meaning get similar numbers, even if they use different words. A small helper model called an embedding model turns each chunk into one of these number lists.
  • Vector store. A vector is just that list of numbers, and a vector store (or vector database) is a database built to find the vectors closest in meaning to a query. Examples include tools such as FAISS, pgvector, or Qdrant. They are named here only as neutral options, not as a required platform.
  • Top-k retrieval. When a question comes in, you embed the question, ask the vector store for the closest chunks, and keep the best handful. The "k" is just how many you keep, so "top-k" means "the top k closest chunks," for example the top 5.
  • Re-ranking. The first search is fast but rough. A re-ranker is a more careful (and slower) model that looks at the question and each candidate chunk together and reorders them by true relevance. You run it only on the handful of candidates, so it stays affordable.

So the naive pipeline is: chunk your docs, embed them, store them, then at query time embed the question, fetch the top-k chunks, paste them into the prompt, and let the model answer.

Why naive search is not enough: hybrid search

Embedding search is great at meaning but weak at exact matches. If a user searches for an error code like ERR_0x4F2 or a product ID like SKU-99812, embeddings often miss it, because the meaning of a random code is not captured well by numbers. The old-fashioned keyword approach handles those perfectly.

The fix is hybrid search: run both at once. You combine semantic search (the embedding-based search above) with lexical search (classic keyword matching). The standard keyword method is called BM25 (Best Match 25, a well-established ranking formula that scores how well a document's exact words match the query). BM25 catches the exact strings that embeddings fluff, and embeddings catch the paraphrases that keywords miss. Practitioners report that adding hybrid search is often the single biggest jump in retrieval quality. (Digital Applied, hybrid search reference)

To merge two ranked lists into one, the sane default is Reciprocal Rank Fusion (RRF), a simple method that combines results by their positions in each list rather than their raw scores. This sidesteps the headache that BM25 scores and embedding scores are on totally different scales. A common, robust setup is to gather a broad set of candidates with hybrid search plus RRF, then run a cross-encoder re-ranker on the top results before handing the best few to the model. Reported gains from adding a re-ranker on top of hybrid retrieval are large. (Laforge, understanding RRF)

RAG, long context, and agentic retrieval

Two questions come up constantly. First: "Context windows keep getting bigger, so can I just paste everything in and skip retrieval?" Mostly no. The amount of data you might want to search grows far faster than context windows do, so you will keep needing a way to fetch only what matters. Pasting in too much also hurts quality and cost, a problem covered in Module 6. The AI engineer Chip Huyen frames retrieval as something that does not go away no matter how large windows get. (Huyen, AI Engineering, Chapter 7)

Second: how does RAG relate to agents? The clean way to see it is that retrieval is just a tool the model can call, exactly like the tools in Module 4. Instead of always stuffing documents in up front, you give the model a search_docs tool and let it pull information when it decides it needs to. This is sometimes called just-in-time or agentic retrieval. A useful pattern here is progressive disclosure: the agent holds lightweight pointers (file paths, links, short queries) and fetches the full text only at the moment it is needed. (Effective context engineering for AI agents)

Building retrieval as a tool also sets you up neatly for the Model Context Protocol (MCP), an open standard for connecting models to tools and data sources, which you will study in Module 9.

Measure the retriever by itself

The most common mistake in RAG is judging the whole system end to end and concluding "RAG is bad" when the real problem is retrieval. If the right chunk never made it into the prompt, no model can answer well. So measure the retriever separately: for a set of test questions, check whether the correct chunks appear in the results (this is roughly what recall measures) and how high they rank (roughly precision). Only once retrieval is solid should you worry about the generation step. Most "RAG is bad" complaints are retrieval problems in disguise.

Lab 5: Build a RAG pipeline over a body of text you care about, such as your own notes or a project's documentation. Start with naive top-k vector search and measure its retrieval quality on a handful of test questions. Then add BM25 plus RRF hybrid search and a re-ranker, and record the improvement in numbers. Finally, refactor retrieval into a tool the model calls on demand, so you see the connection to Module 4 and to agents.

✅ Best practices:

  • Measure retrieval separately from generation, because most failures live in retrieval.
  • Use hybrid search (semantic plus BM25) by default, fused with RRF.
  • Add a re-ranker on the top candidates for a strong, cheap quality boost.
  • Chunk thoughtfully so each passage stands on its own.

⛔ Pitfalls:

  • Believing "RAG is dead" marketing. What is dying is naive single-vector top-k search, not retrieval itself.
  • Only ever testing the full system and never isolating the retriever.
  • Pre-loading everything into the context window and degrading quality (see Module 6).
  • Relying on embeddings alone for exact codes, IDs, and names.

🏁 Milestone: You have a retrieval system whose precision measurably improved after adding hybrid search and re-ranking, and you have exposed it as a tool the model can call.

✎ Publish this: Write a post titled something like "RAG is not dead, naive RAG is," and show the before-and-after numbers from your own pipeline so readers can see the difference for themselves.
