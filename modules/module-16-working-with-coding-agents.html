🎯 Objective: Become fluent at driving a coding agent (a tool that writes and runs code for you, such as an AI pair-programmer in your terminal or editor) to produce reviewed, tested software you would stand behind. This is the meta-skill you will use to build the capstone itself.

Why this module exists: the rest of the course teaches you to build agent systems. This module teaches you to use an agent as your daily tool to build them. The two senses of "agentic engineering" meet here: engineering with agents, and engineering of agents. Much of the practical wisdom below is distilled from working engineers writing openly about the craft, especially Simon Willison's Agentic Engineering Patterns guide and his related posts, which are cited throughout as sources.

Principle 1: agentic engineering vs. "vibe coding"

A useful definition, from Simon Willison, is that agentic engineering is "the practice of developing software with the assistance of coding agents," and it rests on one mechanic: agents run tools in a loop to reach a goal (source). The decisive property of a coding agent is that it can actually execute code, not just suggest it. Without the ability to run the code, the output of an LLM (Large Language Model, the AI that predicts text) is just an untested guess.

It is worth being precise about vibe coding, a phrase coined by Andrej Karpathy. It means prompting for code while you "forget that the code even exists," which is fine for a quick, low-stakes prototype. The term should be reserved for unreviewed, prototype-quality generated code. The moment you start reviewing, testing, and making sure you understand the result, that is not vibe coding any more, it is ordinary software development (source). Do not let the buzzword talk you out of the discipline.

Cheap to write is not the same as good. Writing code is cheap now, but shipping worse code with agents is a choice. Cheaper generation is not a license for lower standards. Treat the time you save as a budget to raise the quality bar instead.

Principle 2: keep a personal library of things you know how to do

One of the most valuable assets you can build as a software professional is a deep, personal collection of solved problems, each one backed by a small piece of running code that proves the answer works. The reasoning is simple: you only ever need to figure out a useful trick once, and from then on you can reuse it. Coding agents make this habit far more powerful, because a documented, runnable example can be adapted to a new situation in seconds. A reliable prompting pattern is to tell an agent to build something new by combining two or more existing working examples (source). Start your own collection now: keep a "TIL" log (TIL means "Today I Learned," a running note of small lessons), save tiny self-contained tools, and keep runnable snippets you can paste back in later.

How coding agents actually work

There is no magic here. A coding agent is just a piece of software that wraps an LLM and gives it extra abilities through hidden instructions and a set of callable tools. It is the same basic building block from earlier in the course (an LLM, plus a system prompt, plus tools, running in a loop), pointed at a shell (the command line) and your files. A few things are worth understanding concretely: text is counted and billed in tokens (small chunks of text); LLMs are stateless, meaning every request starts from a blank slate with no memory of the last one unless you resend it; caching can reuse repeated parts of a prompt to save money and time; there is a hidden system prompt guiding the agent; tool calling is how the model asks to run an action; and "thinking" is just the model spending extra tokens to reason before answering.

The core skill: designing agentic loops

A clear definition of an agent is "something that runs tools in a loop to achieve a goal" (source). The real engineering skill is designing that loop so the agent can check its own work. A load-bearing idea here is that the value you get from a coding agent is hugely amplified by a good, cleanly passing test suite (a set of automated checks that confirm the code does what it should). Agentic loops work best on problems with clear success criteria that reward trial and error, such as fixing a bug, speeding up slow code, or upgrading dependencies, because the agent can keep trying until the tests pass. This is the same verification principle from the autonomous-agents module, turned into a daily habit. It is also a genuinely new skill that has only become practical as models have gotten good enough to drive these loops reliably.

Running agents safely: sandboxes, auto-approve mode, and budgets

An agent with access to your shell has been memorably described as "an LLM wrecking its environment in a loop." So a few safety rules matter:

  • Auto-approve mode (sometimes called "YOLO mode"), where the agent runs commands without asking permission each time, is genuinely more effective, because constant approval prompts slow it down and stop it from solving problems by rapid trial and error. But only run it inside a sandbox (an isolated, throwaway environment). Running the agent inside a container (for example, Docker, a popular tool for isolating software) is a reasonable risk for most people.
  • Contain the blast radius. Give the agent credentials only for test or staging environments where any damage stays contained, and if a credential can spend money, set a tight budget limit (one practitioner created a dedicated cloud account capped at five dollars).
  • This is the lethal trifecta (covered in the safety module) in coding-agent form: never give an autonomous agent your secrets, untrusted input, and unrestricted internet access all at the same time.

Subagents, without over-engineering

A subagent is a fresh copy of the agent with its own clean context window (the limited amount of text a model can consider at once). Subagents are a simple, effective way to handle a larger task without using up the main agent's valuable context: you hand a chunk of work to a subagent, and only its summary comes back. A common use is sending a subagent to explore and map an unfamiliar codebase, and a parent agent can run several subagents in parallel, often on cheaper, faster models. Still, resist the multi-agent hype: a single capable agent is usually perfectly able to debug or review its own output as long as it has enough context to work with, so do not build dozens of specialist subagents just because you can (source).

Testing and QA: the part you cannot outsource

QA means "Quality Assurance," the work of confirming software actually works. This is the part you stay responsible for:

  • Run the tests first. Establish a cleanly passing baseline before you change anything, so you know what "working" looked like to begin with.
  • Use red/green TDD. TDD means "Test-Driven Development": have the agent first write a failing test ("red") that defines the desired behavior, then make it pass ("green"). Writing the test first forces the code to actually be exercised.
  • Do hands-on manual testing. The hard rule: never assume code an LLM wrote works until it has actually run, and remember that passing tests does not guarantee the feature behaves as intended. Drive the real, running software yourself. For example, call the API with a tool like curl (a command-line tool for making web requests), or automate a real browser with a tool such as Playwright. Seeing a feature work with your own eyes before you release it is worth the few minutes it takes (source).

Review discipline and git: you stay accountable

The cardinal anti-pattern is filing a pull request (a "PR," a proposed code change submitted for review) containing code you have not read yourself. If you just relay an agent's output without understanding it, you are adding no value that the next person could not get by prompting an agent themselves. The first review pass is your responsibility and should not be farmed out. Keep PRs small, include context and proof that you tested the change, and review the agent's PR description too, because agents write convincing-looking summaries that can paper over what the code really does. Pair this with good git discipline (git is the standard version-control tool that tracks the history of your code): work on branches, commit often, and tidy the history so the agent's work lands as clear, coherent commits (source).

Running several agents at once

Once you trust your loops, you can run several agents in parallel, but know the real bottleneck: AI-generated code still has to be reviewed, and how fast you can review is the limit. So parallelize the safe work first: research and "scout" tasks that answer questions or make recommendations without changing anything, and tasks that started from a clear specification you wrote (those are cheaper to review). Run riskier work asynchronously and inside a sandbox, so the worst case stays contained (source).

Making it concrete with whatever tool you use. The patterns above apply to any capable coding agent (for example, tools such as Claude Code, Cursor, Aider, or similar). Whichever you pick, the durable habits are the same: give it lasting project context in a small "instructions" file at the root of your project (a context-engineering artifact, covered earlier in the course), use subagents for scoped work, lean on permissions and sandboxing for safe auto-approve runs, and use a scripting interface or SDK (Software Development Kit, a toolkit for building on top of a product) when you want to wire agents into your CI (Continuous Integration, the automated system that builds and tests your code on every change).

Lab 16: Use a coding agent of your choice to build one real component of your capstone end to end: (a) run the tests first for a green baseline; (b) write a failing test for the new behavior; (c) let the agent make it pass inside a sandbox with scoped, budget-limited credentials (auto-approve mode is fine there); (d) do hands-on manual testing, driving the running feature and seeing it work with your own eyes; (e) review every change (and the agent's PR description) before you commit. Keep a short log of where the agent did well and where your review caught problems, and save one reusable snippet to your personal library.

✅ Best practices: design a loop the agent can verify by itself (tests, types, linter, browser); sandbox shell access and cap any spending; run the tests first; review every change and every PR description; keep PRs small; commit often on branches; keep a project instructions file; and save the solutions you work out for reuse.

⛔ Pitfalls: pushing unreviewed code onto collaborators (you are accountable, including for AI-written PR text); trusting a passing test suite as proof the feature truly "works"; running a shell-capable agent outside a sandbox; using auto-approve mode with real credentials or a real budget; over-engineering a swarm of specialist subagents; and assuming the model thinks like a person.

🏁 Milestone: You can drive a coding agent through a verifiable loop to ship a reviewed, tested component you would defend in code review, and you can explain why that is "software development," not "vibe coding."
