🎯 Objective: Become fluent at driving a coding agent (Claude Code) to produce reviewed, tested software you'd stand behind — the meta-skill you'll use to build the capstone itself. This is the curriculum's deepest single integration of one practitioner: it works through Simon Willison's Agentic Engineering Patterns guide and his surrounding writing, because as of 2026 he is the clearest, most honest voice on this exact craft. His ideas also run as a thread through the whole course (you've met them in Modules 1, 3, 8, 12, and 14); here they come together.

Why this module exists: the rest of the curriculum teaches you to build agent systems; this teaches you to use an agent as your daily tool to build them. The two senses of "agentic engineering" meet here. (It's also your current reality, Fumnanya — you already live in Claude Code; this names the craft.)

Principle 1 — Agentic engineering vs. "vibe coding"

Willison defines agentic engineering as "the practice of developing software with the assistance of coding agents," resting on one mechanic: "Agents run tools in a loop to achieve a goal." The decisive property of a coding agent is that it can actually execute code — without execution, LLM output has limited value.

He is deliberately precise about vibe coding (Karpathy's term): it means prompting for code while you "forget that the code even exists" — fine for low-stakes prototypes. He reserves the term for "unreviewed, prototype-quality LLM-generated code," and insists the rest — "reviewing, testing, ensuring comprehension… is not vibe coding, it's software development." Don't let the buzzword dilute the discipline.

Cost ≠ quality. "Writing code is cheap now," but "shipping worse code with agents is a choice. We can choose to ship code that is better instead." Cheaper generation is not a license for lower standards — it's a budget to raise them.

Principle 2 — Hoard things you know how to do

"A key asset to develop as a software professional is a deep collection of answers to questions… accompanied by proof of those answers" — and "the best way to be confident… is to have seen them illustrated by running code." Because "we only ever need to figure out a useful trick once," agents amplify hoarding: documented, runnable examples can be adapted instantly. His favorite prompting pattern is "tell an agent to build something new by combining two or more existing working examples." Start your own hoard — a TIL log, single-file HTML tools, runnable snippets — from day one of the course.

How coding agents actually work

Strip the mystique: "A coding agent is a piece of software that acts as a harness for an LLM, extending that LLM with additional capabilities that are powered by invisible prompts and implemented as callable tools." The core is the same atom from Phase 2 — LLM + system prompt + tools, in a loop — pointed at a shell and filesystem. Things worth understanding concretely (he explains all of them): tokens and per-token billing; chat-templated prompts (and that "LLMs are stateless: every time they execute a prompt they start from the same blank slate"); token/prefix caching; the hidden system prompt; tool calling; and "thinking" as extra reasoning tokens.

The core skill — designing agentic loops

Willison's sharpest contribution: "My preferred definition of an LLM agent is something that runs tools in a loop to achieve a goal." The engineering skill is designing that loop so the agent can verify its own work. His load-bearing claim: "The value you can get from coding agents is massively amplified by a good, cleanly passing test suite." Agentic loops shine on problems with clear success criteria that reward trial and error — debugging, performance optimization, dependency upgrades. This is Module 8's verification principle and the Agent SDK's "verify work" step, made daily practice. And it's "a very new skill" — Claude Code only shipped February 2025.

Running agents safely — sandboxes, YOLO mode, budgets

An agent with shell access is, in Solomon Hykes' quip Willison likes, "an LLM wrecking its environment in a loop." So: - YOLO mode (no approval prompts) is genuinely more effective — constant approvals "dramatically reduce their effectiveness at solving problems through brute force" — but only run it inside a sandbox. "Running agents in Docker or Apple's container tool is a reasonable risk to accept for most people." - Contain the blast radius: "provide credentials to test or staging environments where any damage can be well contained," and "if a credential can spend money, set a tight budget limit" (he made a dedicated cloud org with a $5 cap). - This is the lethal trifecta (Module 14) in coding-agent form: never give an autonomous agent your secrets and untrusted input and unrestricted outbound access at once.

Subagents — without over-engineering

"Subagents provide a simple but effective way to handle larger tasks without burning through too much of the coding agent's valuable top-level context" — a subagent is "a fresh copy… with a new context window." Claude Code's Explore subagent maps an unfamiliar repo; a parent can run several subagents in parallel on cheaper models (e.g. Haiku). But resist the multi-agent hype (echoing Phase 4): "Your root coding agent is perfectly capable of debugging or reviewing its own output provided it has the tokens to spare." Don't build dozens of specialist subagents.

Testing & QA — the part you cannot outsource

  • First run the tests — establish a cleanly passing baseline before you change anything.
  • Red/green TDD — have the agent write a failing test that defines the behavior, then make it pass; test-first forces it to actually exercise the code.
  • Agentic manual testing — his hard rule: "Never assume that code generated by an LLM works until that code has been executed," and crucially "just because code passes tests doesn't mean it works as intended." Drive the running software — curl the API, or browser-automate with Playwright ("the most powerful of these today"). "I like to see a feature working with my own eye before I land it in a release."

Review discipline & git — you stay accountable

The cardinal anti-pattern: "Don't file pull requests with code you haven't reviewed yourself." If you just relay agent output, "they could have prompted an agent themselves. What value are you even providing?" So: "the initial review pass is your responsibility, not something you should farm out to others"; keep PRs small; include context and proof of manual testing; and note that "agents write convincing looking pull request descriptions — you need to review these too." Pair this with git discipline: branches/worktrees for background runs, frequent commits, and rewriting history so agent work lands as coherent commits.

The parallel coding-agent lifestyle

Once you trust your loops, you can run several agents at once — but know the real constraint: "AI-generated code needs to be reviewed, which means the natural bottleneck… is how fast I can review the results." So parallelize the safe work: research/scout tasks that "answer questions or provide recommendations without making modifications," and tasks "that started from your own specification" (cheaper to review). Run risky ones async/sandboxed so "the worst that can happen is my source code getting leaked." He notes models "only really got good enough to drive… in the past few months — Claude 4 and GPT-5 in particular."

Claude layer: Claude Code is the reference coding agent for every pattern above. Give it durable project context via a CLAUDE.md file (a context-engineering artifact — Module 3), use subagents (incl. Explore) for scoped work, hooks/permissions/sandboxing for safe YOLO runs, and the Claude Agent SDK to script agents into CI. (Claude Agent SDK)

Lab 16: Use Claude Code to build one real component of your capstone end-to-end, Willison-style: (a) first run the tests for a green baseline; (b) write a failing test for the new behavior; (c) let the agent make it pass inside a sandbox with scoped, budget-limited credentials (YOLO mode is fine there); (d) do agentic manual testing — drive the running feature and see it work with your own eyes; (e) review every diff (and the agent's PR description) before you commit. Keep a short log of where the agent excelled and where your review caught problems. Add one reusable snippet to your personal "hoard."

✅ Best practices: design a loop the agent can verify itself (tests/types/linter/browser); sandbox shell access and cap any spend; run tests first; review every diff and PR description; keep PRs small; commit often on branches; maintain a CLAUDE.md; hoard what you solve.

⛔ Pitfalls: inflicting unreviewed code on collaborators (the cardinal anti-pattern — you're accountable, including for AI-written PR text); trusting a green test suite as proof it "works"; running a shell-capable agent outside a sandbox; YOLO mode with real credentials/budget; over-engineering a swarm of specialist subagents; anthropomorphizing failures.

🏁 Milestone: You can drive a coding agent through a verifiable loop to ship a reviewed, tested component you'd defend in code review — and articulate why that's "software development," not "vibe coding."
