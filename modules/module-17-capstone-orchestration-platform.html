🎯 Objective: Compose everything you have learned into a single, real, evaluated, observable, and hardened agent orchestration platform (a system where one coordinating agent plans the work and delegates it to other agents and tools), and document the build so others can follow it.

This is the capstone, the project that proves the whole course. There is no new concept to learn here. The skill on display is integration, which means making all the separate pieces work together as one dependable system. Integration is its own discipline, and it is exactly what enterprise teams hire agentic engineers to do. Build it using the coding-agent craft from Module 16.

Pick a real, non-trivial problem worth automating, then hold yourself to the bar below. Each item points back to the module where the skill was taught.

Required architecture (the exit bar)

  1. Orchestration. A lead "orchestrator" agent that breaks a complex task into smaller pieces and delegates them to specialized worker agents (Modules 7 and 10), together with a deliberate, cost-aware decision about whether you even need more than one agent (Module 10).
  2. Tools via MCP. Real capabilities exposed through MCP (Model Context Protocol, the open standard for connecting agents to tools and data), including at least one tool server you built yourself (Modules 4 and 9).
  3. Retrieval and memory. Hybrid retrieval (combining keyword search and meaning-based search) offered to the agent as a tool, plus a way to compact or store information so the system stays coherent across long tasks (Modules 5 and 6).
  4. The agent loop done right. A bounded loop (a capped number of steps), verification or self-check steps, and sensible stopping conditions so the agent knows when it is finished or stuck (Module 8).
  5. An evaluation suite. A set of automated tests built from real failures you observed, reporting a pass rate for each component so you can prove it works and catch regressions later (Module 12).
  6. Observability. Full tracing of every step, with the cost and the latency (wait time) recorded per step, so you can see inside the system while it runs in production (Module 13).
  7. Safety. Least privilege (give each part only the access it truly needs), a human approval step before any consequential action, and a written defense against the lethal trifecta (Module 14).
  8. Hardening. A per-task budget for cost and latency, the right model size for each job, caching where it helps, and graceful fallbacks when something fails (Module 15).

Choosing a domain. Pick something you would actually use, because real use surfaces real problems. Some ambitious, enterprise-flavored options:

  • A research orchestrator that answers deep questions with cited, synthesized reports. This is a breadth-first task, which is the sweet spot for using several agents at once.
  • A customer-operations resolver that triages incoming requests, gathers context from internal systems, drafts a resolution, and routes anything risky to a human.
  • An incident-response assistant for software teams that reads logs and alerts, proposes a diagnosis, and prepares a fix for a human to review.
  • A data-analysis agent that turns plain-language questions into database queries, runs them safely, and explains the results.

Capstone deliverables (what "done" looks like):

  1. The working system, in a code repository.
  2. A system-design document that explains each architectural choice and the trade-off behind it, especially single-agent vs multi-agent and framework vs no framework.
  3. An evaluation report with pass rates per component.
  4. A cost and latency report per task.
  5. A security note documenting your lethal-trifecta audit and how you mitigated it.
  6. A build-in-public write-up, one post per phase, so others can learn from the journey.

🏁 Final milestone. You are an agent orchestrator when you can: stand up a system that splits work across agents, ground it in tools and data via MCP, manage context across long tasks, prove its reliability with evaluations and traces, defend it against prompt injection, state its cost per task, and explain every design decision along with the alternative you rejected. That is not a hobby project. That is the job.

✎ Publish this: the whole capstone, as a series. "I taught myself agentic engineering and built an enterprise-grade agent orchestration platform. Here is everything, including what broke and how I fixed it." The series itself is the proof of the skill.
