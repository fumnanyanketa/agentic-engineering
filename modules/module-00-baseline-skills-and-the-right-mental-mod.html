🎯 Objective: Check the handful of skills you actually need before Phase 1, see what you can safely skip, and install one durable way of thinking: your job is to build a good workplace for a capable but fallible helper, not to program the model line by line.

This module is short on purpose. It is a starting line, not a hill to climb. The goal is to make sure nothing trips you up later and to hand you one idea that the rest of the course leans on. Two terms up front. An LLM is a Large Language Model, the kind of AI that powers chat assistants and writes text one small piece at a time. An agent is an LLM that has been given the ability to take actions in a loop (call tools, read results, decide what to do next) instead of just answering once. Everything in this course builds toward agents.

Prerequisites you genuinely need

Close any gaps here before you start Phase 1. None of these require deep expertise. You need working comfort, not mastery.

  • Python. This is the common language of AI engineering, and it shows up in nearly every job posting in the field. If you can read and write everyday Python (functions, loops, dictionaries, importing libraries), you are ready. (Python to AI Engineer roadmap)
  • Async Python (asyncio). "Async" means running several slow tasks at the same time instead of one after another. You will often send many model requests at once, and a model request is slow (you are waiting on a network call). Code that handles one request at a time (called "blocking" code, because each call blocks the next) will not keep up. You only need the basics: async def, await, and running several calls together.
  • Type hints and Pydantic. Type hints are small notes in Python that say what kind of value a variable holds, such as name: str. Pydantic is a popular library that uses those hints to check data and reject anything in the wrong shape. You will lean on it constantly to make sure the model's output is valid and machine-readable rather than a loose blob of text. This is load-bearing, not cosmetic. (Pydantic docs)
  • HTTP, REST, and streaming. Model providers are reached through an API (Application Programming Interface, a way for your code to talk to a remote service). These are ordinary web APIs. HTTP is the protocol your code uses to make the request; REST is the common style those requests follow. Streaming means the reply arrives token by token as it is generated, instead of all at once at the end. You should understand sending a request, reading a streamed response, retrying when a call fails, and respecting rate limits (caps on how many requests you may send in a window of time).
  • Git plus basic testing. Git is the standard tool for version control, which means keeping a tracked history of every change. Version everything, including your prompts, because a prompt is a piece of software too. Get in the habit of writing small tests for your ordinary, non-AI code first, so you have a stable base before you add the unpredictable part. (Chip Huyen, AI Engineering open book)

What you can defer or skip

You are becoming an engineer who builds on top of existing models, not a researcher who builds the models themselves. A foundation model is a large pretrained model (like the LLMs this course uses) that you adapt to your needs rather than train from scratch. Because you build on these, you can safely set the following aside for now:

  • Training a model from scratch.
  • GPU kernel work (low-level code that squeezes performance out of the graphics chips used to train models).
  • The heavy math, such as backpropagation derivations (the calculus that adjusts a model during training) and optimization theory.

Knowing this material is not harmful, but it is not on the path to shipping useful agentic systems. Spend your early energy elsewhere. (Chip Huyen, AI Engineering open book)

The one mental model: give the model a good workplace

Here is the idea to carry through the entire course. You are not programming the model. You are engineering its context and its environment so that a capable but fallible reasoner can succeed. Think of the model as a smart, fast new teammate who has no memory of yesterday, cannot see your screen, and will confidently guess when information is missing. Your job is to set up that teammate's desk: the right instructions, the right reference material, the right tools, and a way to check the work.

This reframing matters because almost everything later in the course is one facet of this single job. Prompting is how you give clear instructions. Context engineering is what reference material you place on the desk. Tool use is what equipment you hand over. Memory and retrieval are how the desk stays stocked over time. Evaluation is how you check the output. When something goes wrong, the first question is rarely "is the model broken?" It is usually "what was missing or confusing in the workplace I built?" This view is the through-line of recent practitioner and lab writing on getting good results from models, which keeps returning to the same point: most of the craft is managing what the model can see and do. (Simon Willison, how I use LLMs to write code)

Lab 0: Set up a clean workspace. Install a recent Python (3.10 or newer), create a virtual environment (an isolated folder for this project's packages), and install the official client library for any one model provider you choose. Put your API key (the secret token that authorizes your requests) in an environment variable, never written directly in your code. Then write a small "hello, model" script that sends one prompt, streams the reply to your screen as it arrives, and prints the token usage (a token is a small chunk of text, and tokens are how usage and cost are measured). Commit it to Git. Finally, define one Pydantic model describing a simple data shape and have the API return JSON that validates against it. If validation passes, you have proven the full loop: authenticate, call, stream, and get a checked, structured result.

✅ Best practices:

  • Keep secrets such as API keys in environment variables or a secrets manager, never in source code or Git history.
  • Watch your token counts from day one. They are both your bill and your context budget.
  • Pin your dependency versions so a project that works today still works next month.
  • Write tests for the predictable, non-AI parts of your code before you add the unpredictable model calls.

⛔ Pitfalls:

  • Hardcoding API keys, which leaks them the moment you push to a shared repository.
  • Ignoring token counts until the bill or a context-limit error surprises you.
  • Skipping async early and then hitting a wall once you need many concurrent calls in later phases.
  • Treating the model as a normal program that should behave the same way every time. It will not.

🏁 Milestone: You have a clean repository, a working authenticated call that streams and reports token usage, and a structured output that passes Pydantic validation. You can also state, in one sentence, the "good workplace" mental model in your own words.
