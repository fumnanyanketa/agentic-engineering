🎯 Objective: Reliably get good behavior out of a single model call. This is a genuine skill, but it sits inside the larger job of building a good workplace for the model. Treat it as a craft, not the whole discipline.

A prompt is simply the text you send to the model. Prompt engineering is the practice of shaping that text so the model does what you want, consistently, not just once by luck. The principles below are vendor-neutral, meaning they hold no matter which provider's model you use, because they come from how all of these models read and continue text.

The core techniques

Most reliable prompting comes down to a small set of moves. You will combine them, not pick just one.

  • Be clear, direct, and specific. The model cannot read your mind and will fill any gap with a guess. Say exactly what you want, in what format, for whom, and what to avoid. "Summarize this in three bullet points for a busy manager, no jargon" beats "summarize this."
  • Few-shot examples (show, do not only tell). A few-shot prompt includes a few worked examples of input and the desired output, so the model can copy the pattern. ("Zero-shot" means no examples; "few-shot" means a handful.) Examples are often the single most effective way to lock in tone and format, especially for tasks like classification, extraction, or translation. (prompt engineering techniques overview)
  • Chain-of-thought for reasoning tasks. Chain-of-thought (CoT) means asking the model to work through its steps before giving a final answer, for example by adding "think step by step." This helps on math, logic, and multi-step decisions because it gives the model room to reason instead of jumping straight to a guess. Note that some newer "reasoning" models already do this internally, so you may not need to ask. (chain-of-thought, ReAct and few-shot guide)
  • Role and system prompts. A system prompt is a separate, higher-priority instruction that sets the model's role and standing rules for the whole conversation, such as "You are a careful financial analyst. Never invent numbers." It frames everything the user then says.
  • Clear structure for mixed content. When a prompt contains several different things at once (instructions, reference text, examples), wrap each part in clearly labeled sections so the model does not confuse them. Many practitioners use lightweight tags such as <instructions>, <context>, and <examples>, or plain headings. The point is unambiguous boundaries, and this technique works across providers.
  • Output formatting and structured outputs. If your code will read the answer, ask for a strict, machine-readable format such as JSON, and describe the exact fields. A structured output is a response that follows a fixed schema (a defined shape) so your program can parse it reliably instead of scraping free text.
  • Iterate against examples, not vibes. Do not judge a prompt by reading one nice answer. Collect real example inputs with the outputs you expect, and tune the prompt until it passes them.

A useful reality check on the field: prompting has matured from a bag of tricks into a measurable practice, and even simple, low-effort techniques like zero-shot chain-of-thought remain widely used precisely because they are cheap and reliable. The fancier methods (tree-of-thought, graph-of-thought, and so on) exist for genuinely hard branching problems, but you will get most of your wins from the basics above. (prompt engineering best practices)

When the task is hard: more reasoning room

For difficult problems, give the model more room to reason before it commits. Two vendor-neutral levers help. First, explicit chain-of-thought, as above. Second, many providers now offer a "thinking" or "reasoning" mode where the model spends extra internal effort (and extra tokens) on hard questions; turn it on for genuinely complex tasks and leave it off for simple ones, since it costs more and adds latency (the wait time for a response). The durable principle is the same whatever the feature is called: match the amount of reasoning effort to the difficulty of the task.

Lab 2: Pick a real task, for example "extract structured fields (date, amount, vendor) from a messy invoice or email." Write a first version (v1) of your prompt. Then build a small test set of about 15 example inputs, each paired with the output you expect. Run your prompt against all 15, count how many it gets right, and improve the prompt until it passes. Change only one thing at a time so you can tell what helped. Keep that test set. It is the seed of your evaluation portfolio (your growing collection of tests, which later modules build on).

✅ Best practices:

  • Version your prompts in Git, the same way you version code.
  • Change one variable at a time so you can attribute any change in quality to a specific edit.
  • Let examples carry the load. A good example often replaces a paragraph of instructions.
  • Ask for structured output whenever a program will consume the result.

⛔ Pitfalls:

  • "Prompt and pray": shipping a prompt with no test set, then being surprised when it fails on new inputs.
  • Over-instructing until the prompt is so rigid it breaks on anything slightly unexpected.
  • Treating prompt engineering as the entire discipline. It is one part of building the model's workplace.

🏁 Milestone: You have a prompt that passes a held-out example set, meaning examples you did not look at while tuning. Passing tests you tuned on proves little; passing fresh ones shows the prompt actually generalizes.

✎ Publish this: Write a short post titled "Prompt engineering is not dead, it just got demoted." Explain where prompting genuinely matters, where it stops being enough, and show one before-and-after prompt with the test numbers that prove the improvement.
