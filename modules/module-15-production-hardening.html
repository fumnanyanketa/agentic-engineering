🎯 Objective: Make your system survivable in the real world. By the end you can take a working agent and make it noticeably cheaper and faster without losing quality, and you can keep it running when an outside service fails.

A demo only has to work once, for you, on a good day. A production system has to work for many people, over and over, without costing too much, making them wait too long, or falling over when something outside your control breaks. Production hardening means turning that fragile demo into something dependable. This module covers the three pressures that decide whether an agent survives contact with real users: cost, latency, and reliability. (As a reminder, latency just means how long a user waits for a response.)

Why reliability is harder for agents

An agent works in steps: think, call a tool, read the result, think again, repeat. Each step has some chance of going wrong, and the errors compound. If a single step is 95% reliable, that sounds great, but ten steps in a row is roughly 0.95 multiplied by itself ten times, which is about 60%. Twenty steps drops below 40%. This is the same compounding math introduced back in Module 8. Small per-step error rates become large whole-task failure rates as the chain gets longer (Huyen, Building a GenAI platform).

You fight this in a few durable ways, none of which depend on any particular vendor:

  • Shorter loops. Fewer steps means fewer chances to fail. Cut any step that does not earn its place.
  • Verification steps. Have the agent (or a cheap separate check) confirm a result before relying on it, so an error is caught early instead of poisoning later steps.
  • Consensus or voting, where it pays. For a high-stakes decision, run a step more than once and take the answer that agrees with itself. This costs more, so use it only where the extra reliability is worth the extra spend.
  • Retries with backoff. If a step fails, try again. "Backoff" means waiting a little longer before each retry (say 1 second, then 2, then 4) so you do not hammer a service that is already struggling.

Cost levers: spend less per task

Every token in and out of a model costs money, and agents are token-hungry because they loop. As a rough rule of thumb, an agent can use several times the tokens of a single chat answer, and a multi-agent system can use many times more, because the agents talk to each other and re-read context repeatedly. So cost control is not optional. Here are the main levers, all vendor-neutral.

Right-size the model to the task

Most model providers offer a range of tiers, from small and cheap and fast, up to large and expensive and slow. The instinct to always reach for the biggest model is a costly mistake. Match each sub-task to the smallest tier that still passes its quality bar. A simple classification step might run fine on a tiny model, while only the hardest reasoning step needs the top tier. You decide this by measuring (using the evaluations from Module 12), not by guessing.

Prompt caching

Prompt caching is one of the biggest savings available, and it works because of how models process text. A model first reads (the technical word is "prefill") all the input before it starts answering. If the start of your prompt is the same across many requests (your instructions, your tool definitions, a reference document), the model can save the work it did on that shared opening and reuse it next time instead of re-reading it from scratch. Providers that offer this report large savings on the cached portion: on long, stable prefixes, roughly up to 90% lower cost and up to 85% lower time-to-first-token on the reused part (Redis, what is prompt caching).

To benefit, order your prompt from most stable to least stable: fixed instructions and tool definitions first, reference material next, and the parts that change every time (the live user question, any fresh data) at the very end. The reason is that changing any part of the prompt usually invalidates the cache for that part and everything after it, so you want the changing bits last. One team raised its cache hit rate from 7% to 84%, and cut overall cost by more than half, simply by moving frequently-changing data out of the fixed instructions and down to the end of the prompt (ProjectDiscovery, cutting cost with prompt caching).

Context economy and response caching

Because every token is money, trimming unnecessary context (old conversation, oversized documents) directly lowers cost and often improves quality too. Two more caching ideas help when the same questions recur. Exact caching stores the answer to an identical request and replays it, which is safe and simple. Semantic caching tries to reuse an answer for a question that merely means the same thing, judged by similarity. Semantic caching is more dubious: if the similarity check is wrong, you serve a confidently wrong answer to a slightly different question. Use exact caching freely, and treat semantic caching with caution (Huyen, Building a GenAI platform).

A model gateway: one front door

A model gateway is a single piece of your own software that sits between your application and the model providers, so every model call goes through one place. Routing all calls through one door buys you three things at once:

  • Access control. Your secret API keys (the credentials that authorize and bill your model calls) live in the gateway, not scattered through your code, which reduces the chance one leaks.
  • Fallback policies. If one provider is down, rate-limited, or slow, the gateway can automatically retry or switch to a backup model so your users still get an answer. ("Rate-limited" means the provider is temporarily refusing extra requests because you have sent too many too fast.)
  • A natural home for caching. Since every call passes through it, the gateway is the obvious place to add the caching above.

This pattern keeps you portable: because your app talks to your gateway instead of directly to one vendor, swapping providers is a change in one place (Huyen, Building a GenAI platform).

Human-in-the-loop: the final backstop

Human-in-the-loop means a person reviews or approves certain actions before they take effect. It is your last line of defense for anything consequential or anything the agent is unsure about. Tie it to confidence and stakes: let the agent act on its own for low-risk, high-confidence steps, and require a human to approve anything irreversible or expensive. This is the reliability counterpart to the security gating you saw in Module 14.

Lab 15: Harden the agent you have been building. Do all four and report numbers. (1) Downshift every sub-task to the cheapest model tier that still passes its evaluation from Module 12 (do not skip the re-check). (2) Add prompt caching by reordering your prompt so the stable parts come first and the changing parts come last. (3) Add retries with backoff plus a fallback to a backup model when a call fails. (4) Report before-and-after numbers for three things together: cost per task, latency per task, and pass rate. The pass rate is the proof that you made it cheaper and faster without quietly trading away quality.

✅ Best practices:

  • Treat cost and latency per task as first-class metrics you measure, not afterthoughts.
  • Use the cheapest model tier that still passes the task's evaluation.
  • Cache aggressively: exact caching freely, prompt caching by ordering stable content first.
  • Fail gracefully with retries, backoff, and a fallback model behind a gateway.
  • Keep a human in the loop for consequential or low-confidence actions.

⛔ Pitfalls:

  • Reaching for the biggest, most expensive model everywhere by default.
  • No caching, so you pay full price to re-read the same instructions on every request.
  • No fallback path, so one provider outage takes your whole product down.
  • Optimizing cost without re-running your evaluations, which lets quality quietly regress while you celebrate the lower bill.

🏁 Milestone: You have a hardened agent with a documented before-and-after on cost per task, latency per task, and pass rate, proving you cut cost and latency while holding quality steady.
