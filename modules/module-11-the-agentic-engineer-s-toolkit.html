🎯 Objective: Map the tooling landscape and form a defensible view on the "framework or no framework" question. The goal is to choose tools by the shape of your problem, not by what is popular this month.

One definition up front. A framework here means a ready-made software library that handles the plumbing of an agent for you: the loop where the model thinks, calls a tool, reads the result, and thinks again. Without a framework you write that loop yourself by calling a model's API (an API, an Application Programming Interface, is just the way one program talks to another) and wiring up the tools by hand. With a framework, a lot of that is done for you. Both are valid. This module is about choosing on purpose.

The honest meta-point: you can start without a framework

A widely cited guide from one AI lab recommends starting without a framework. Its reasoning is plain: "many patterns can be implemented in a few lines of code," and frameworks "often create extra layers of abstraction that can obscure the underlying prompts and responses, making them harder to debug." In other words, a framework can hide the very things you most need to see when something goes wrong: the exact prompt sent and the exact response received. The same guide adds that if you do adopt a framework, you should understand the code underneath it, because wrong assumptions about what it does for you are a common source of bugs. (an analysis of the "building effective agents" guidance)

This is not an argument that frameworks are bad. It is an argument about order. Once you understand the loop yourself (you have built one by hand in earlier modules), a framework becomes a convenience you reach for deliberately, not a crutch you depend on without understanding.

When a framework helps, and when it gets in the way

A fair, even-handed way to decide:

  • Reasons to use one: it saves you from re-writing common plumbing (retries, state, branching, human-approval steps); it gives your team a shared vocabulary; and the good ones come with built-in tracing so you can see what happened.
  • Reasons to skip one: your task is simple enough to write in a few lines; you need full visibility into every prompt and response; or you want to avoid lock-in (becoming so tied to one tool that switching later is painful).

The landscape (neutral examples, chosen by problem shape)

Treat the names below as interchangeable examples. The right pick depends on your problem, not on a leaderboard, and the field moves fast, so verify current features yourself. (one comparison of agent frameworks)

  • Explicit stateful graphs. Some libraries model your agent as a graph with cycles, branching, retries, saved checkpoints, and human-approval steps. Good for complex flows where you want tight control over the path.
  • Role-based multi-agent. Some let you define agents as named roles (for example "planner," "researcher," "writer") that hand work to each other.
  • Conversation-style multi-agent. Some model the system as several agents talking to each other, often running at the same time.
  • Provider-native agent runtimes. Most major model providers ship their own agent library that fits neatly into their own ecosystem.
  • Type-safe, model-agnostic libraries. Some emphasize strong typing and work across providers, and connect natively to MCP (the Model Context Protocol, an open standard for plugging tools and data into models).
  • Minimal, code-driven libraries. Some stay deliberately small and lightweight, close to writing the loop yourself.

Adjacent tooling (know the categories, not the brand names)

Around the agent itself you will meet two more categories. You do not need to memorize products; you need to know the categories exist.

  • Vector databases store text as numeric vectors so you can search by meaning (used for retrieval, covered in the retrieval module). Options range from tiny ones meant for prototyping, to extensions that live inside a database you already run, to fully managed cloud services, to large self-hosted systems. (an overview of vector databases)
  • Evaluation and observability tools help you measure quality and watch what your agent did. These are covered in Phase 5.

How to choose, in one sentence

Learn the raw loop first; then, if your problem is complex enough that a framework's plumbing clearly earns its keep, pick the one whose shape matches your problem, and keep the underlying prompts and traces visible no matter what you choose.

Lab 11: Take the agent you built by hand in the autonomous-agents module and re-implement it in one framework of your choice. Then write a short, honest comparison covering three things: what the framework saved you, what it hid from you (could you still see every prompt and response?), and when you would choose raw API over a framework for a future project.

✅ Best practices:

  • Learn the primitives (the raw loop) before adopting a framework.
  • Pick a framework by the shape of your problem, not by popularity.
  • Keep the underlying prompts and traces visible whatever you use.

⛔ Pitfalls:

  • Adopting a heavy framework before you understand the loop it is hiding.
  • Framework lock-in that makes switching later painful.
  • Copying a trendy tool without asking whether it fits your problem.
  • Trusting comparison posts uncritically: many are written by the tool's own vendor, and features change fast, so verify yourself.

🏁 Milestone: The same agent, built once by hand and once with a framework, plus a clear-eyed written trade-off explaining what you gained and what you gave up.

✎ Publish this: Write a short piece titled "Do you need an agent framework? I built the same agent with and without one." Use your Lab 11 notes, and be specific about what the framework saved and what it hid.
