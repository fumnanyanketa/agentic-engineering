🎯 Objective: Move from "writing one good prompt" to curating everything the model can see while it works. This reframing is the conceptual hinge of the whole course, and it is what makes the later agent modules possible.

In Module 2 you shaped the words you send. Here you widen the lens to the model's entire context: the full set of text the model can see in a single request, measured in tokens (small chunks of text). Context includes the system prompt, the user's message, any reference documents, examples, past conversation, and the results of any tools the model has used. Context engineering is the practice of deciding what goes into that limited space and what stays out.

What context engineering is

A working definition: context engineering is the set of strategies for choosing and maintaining the best possible set of tokens (information) while the model runs. The key word is best, not most. Context is a finite resource. The model has a fixed context limit, and, just as importantly, it has a limited "attention budget," meaning its ability to focus is spread thinner as you add more text. The analogy practitioners use is human working memory: you can only hold so much in mind at once before details start slipping. (Firecrawl, context engineering · token optimization and the attention budget)

This gives a clean way to see how prompting and context engineering relate. Prompt engineering controls how the model communicates: the phrasing and instructions. Context engineering controls what the model knows: the information available to it. Prompting is now best understood as one part of the bigger task.

Why it grew beyond prompting

Single answers came from single prompts. But agents run over many turns and longer stretches of work, calling tools and accumulating results as they go. That means the context is not a fixed string you write once; it is an evolving state you have to manage as the work proceeds. Old material has to be trimmed, fresh material has to be pulled in, and the whole thing has to stay within budget. Managing that moving target is exactly why context engineering became its own discipline. (Sourcegraph, a practical guide to context engineering)

Context rot: bigger is not a free lunch

Context rot is the measurable tendency of a model to recall and reason less accurately as its context grows. The cause is the limited attention budget: the more tokens you add, the more relationships the model has to weigh, and its focus on any one detail gets diluted. (context engineering and token optimization)

This shows up in "needle in a haystack" tests, where a single fact is buried in a long document and the model is asked to find it. As the document grows, recall drops. Independent benchmarking has found many models falling well below their short-context accuracy once the input gets large; the exact numbers vary by model and are always changing, but the direction is consistent and real. The practical lesson is durable even as the figures shift: a large context window lets you add more text, but adding more text often makes quality worse, not better. (context rot and the attention budget)

The guiding heuristic

Aim for the smallest set of high-signal tokens that still gets the job done. Every token should earn its place. A related idea is writing system prompts at the "right altitude": specific enough to steer the model clearly, but not so packed with rigid detail that there is no room to adapt. Too vague and the model wanders; too rigid and it breaks on anything unexpected. (Sourcegraph, context engineering guide)

This is the same point that experienced practitioners keep arriving at independently: most of the craft of getting good results from a model comes down to managing its context. When working engineers and the labs converge on that sentence from different directions, it is a sign this module is load-bearing rather than a passing trend. (Simon Willison, how I use LLMs to write code)

One concrete lever: caching the stable parts

A vendor-neutral technique worth knowing early is prompt caching. If the beginning of your context is long and does not change between requests (a big system prompt, a fixed reference document), many providers let you cache that stable prefix so it does not have to be reprocessed every time. This is mainly an economy tool: it can sharply cut both cost and the latency (wait time) of long, repetitive requests. The savings vary by provider, so treat any specific percentage as a moving target and check current docs. (context engineering in practice)

Lab 3: Take the task from Module 2. First, deliberately bloat the context: paste in 20,000 or more tokens of only loosely related material around your real input, and watch quality drop. That drop is context rot, and you just produced it on purpose. Then curate: cut the context down to only the high-signal tokens the task actually needs, rerun your Module 2 test set, and measure the recovery. Record both scores. You will have felt the problem and fixed it with your own numbers.

✅ Best practices:

  • Make every token earn its place. If removing something does not hurt the result, leave it out.
  • Cache long, stable prefixes to save cost and latency on repeated calls.
  • Measure recall on your own data instead of assuming a big context window means the model is using all of it.
  • Pull information in just-in-time when the task needs it, rather than front-loading everything.

⛔ Pitfalls:

  • "Just stuff the whole document in." More text frequently lowers quality through context rot.
  • Confusing a large context window with good context engineering. The window is capacity; engineering is what you choose to put in it.
  • Letting an agent's context grow unbounded across turns with no trimming or summarizing.

🏁 Milestone: You can explain, in your own words, the difference between prompt engineering and context engineering, and you can demonstrate context rot with measurements you produced yourself.
