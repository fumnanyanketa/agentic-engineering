Tool use and function calling

A model on its own can only produce text. Tools are how you let it actually do things: search the web, query a database, send an email, run code. This module teaches you to design tools a model can use reliably.

🎯 Objective: Learn how to let a model act on the world through tools, and learn to design those tools so the model uses them reliably. By the end you will have built the full loop yourself, with no framework hiding the moving parts.

A plain language model can only do one thing: produce text. It cannot look up today's weather, read a file, or add two large numbers with guaranteed accuracy. Tool use (also called function calling) fixes that. You give the model a menu of actions it is allowed to take, and when it decides an action would help, it asks for that action by name. Your code runs the action and hands the result back. The model then continues with that fresh information.

The two terms mean the same thing. "Function calling" is the older phrase from regular programming, where a function is a named piece of code you can run. "Tool use" is the newer phrase that most teams now prefer, because the thing on the other end does not have to be a function. It can be an entire web service. We will use "tool" throughout.

What a tool actually is

A tool is just three things bundled together:

  • A name, such as get_weather. The model uses this to refer to the tool.
  • A description in plain English that tells the model what the tool does and when to use it. This is the most underrated part. The model reads it the way a new employee reads a one-line job description, so write it carefully.
  • A schema, which is a structured list of the inputs the tool needs and the outputs it returns. "Schema" just means "a precise description of the shape of some data." The usual format is JSON Schema (JSON, or JavaScript Object Notation, is a simple text format for structured data). For get_weather the input schema might say it needs a city (text) and an optional units (either "celsius" or "fahrenheit").

You hand this bundle to the model along with the user's message. Behind the scenes, the model's makers expose tool calling as a standard part of the request, so the same idea works across providers even though the exact field names differ. The broad pattern of declaring tools with a JSON Schema and letting the model request them has become the de-facto industry approach. (Zylos Research, tool use standards and benchmarks)

The tool-use loop, step by step

This is the single most important thing to understand in this module, so we will walk it slowly. Imagine a user asks, "What is the weather in Paris?"

  1. You send the user's message plus your tool definitions to the model.
  2. The model does not answer directly. Instead it returns a tool call: a structured request that says, in effect, "please run get_weather with city = "Paris"." Note that the model does not run anything. It only asks.
  3. Your code reads that request, runs the real get_weather function (which calls a weather service), and gets back, say, 18C and cloudy.
  4. You send that result back to the model as a tool result, attached to the same conversation.
  5. Now the model has the fact it needed, so it writes the final answer: "It is 18 degrees Celsius and cloudy in Paris."

The key insight: the model and your code take turns. The model decides what to do and your code decides how to do it. A model may also chain several tool calls in a row before answering, looping through steps 2 to 4 multiple times. This back-and-forth is the heart of every agent you will build later.

Errors are results, not crashes

When a tool fails (the weather service is down, the city was misspelled), do not let your program crash and do not silently hide the failure. Instead, send the error back to the model as a tool result, clearly marked as an error, for example: "Error: no city named 'Pariss' found. Did you mean 'Paris'?" Given that message, a capable model will usually correct itself and try again. If you swallow the error, the model is left guessing and the whole interaction degrades. Returning a clear, recoverable error message is one of the highest-leverage things you can do for reliability. (Yan et al., structured reflection for reliable tool interactions, 2025)

Tool design matters as much as the prompt

Beginners assume a weak agent means a weak model. Far more often, the tools are the problem. The model can only be as good as the menu you hand it. Anthropic's engineering team reported spending more time tuning the tools than the prompt for one of their coding agents, and they recommend designing tools to be poka-yoke, a manufacturing term meaning "mistake-proof": shape the inputs so the easy path is the correct path and wrong calls are hard to make. (Building Effective Agents)

A few durable rules of thumb hold across every provider:

  • Consolidate, do not mirror your database. One well-named schedule_meeting tool beats three thin tools (list_users, list_calendars, create_event) that force the model to orchestrate them by hand. More tools is not better. Each extra tool is one more thing the model can pick wrongly.
  • Namespace your tool names so they read clearly, such as github_list_pull_requests or stripe_create_refund. This helps the model tell similar tools apart.
  • Return high-signal output. Give back human-readable identifiers rather than long random codes, and paginate, filter, or trim large results. Every token you return is a token that competes for the model's limited attention and costs money.
  • Write the description and input names for the model, not your backend. Name a field customer_email, not cust_eml_2. The model reads these names as hints.

(Writing tools for agents)

Making the model call tools correctly

Two settings show up under different names across providers but mean the same things everywhere:

  • Tool choice controls whether the model may call a tool, must call some tool, or must call one specific tool. Use "must call a tool" when an answer without one would be useless (for example, a lookup assistant that should never guess).
  • Strict schema validation forces the model's tool inputs to match your schema exactly. Research and provider docs agree that argument correctness, not the number of tools, is what makes or breaks reliability, so turning on strict validation where it is offered is usually worth it. (Symflower, function calling in LLM agents)

One security note to plant early, covered in depth in Module 14: a tool is a door into your systems. Treat every input the model sends to a tool as untrusted, and validate and limit it at the tool itself, never relying on the prompt alone to keep the model in bounds. Prompt injection (tricking the model into misusing a tool) is a leading risk for tool-using systems, so the safe pattern is to enforce limits in code. (Zylos Research)

Lab 4: Give a model two or three real tools, for example a calculator, a web search call, and a small database lookup. Write the full loop yourself with the raw provider request (no agent framework yet, because you must see the turn-taking with your own eyes). Make one tool deliberately bad: a vague description, cryptic input names, and a giant unfiltered output. Watch the model stumble. Then fix that tool using the rules above (clear description, clear input names, trimmed output) and watch the behavior improve. Finally, force one tool to fail and confirm the model recovers when you return the error as a tool result.

✅ Best practices:

  • Design tools for how the model thinks, not for how your backend happens to be structured.
  • Test each tool on its own before wiring it into the loop, so you know failures come from the model's choices, not from a broken tool.
  • Return errors to the model as clear, recoverable messages.
  • Keep tool outputs short and high-signal to protect the context window.

⛔ Pitfalls:

  • Exposing dozens of thin tools that each do one tiny database operation.
  • Cryptic tool output that floods the context window with noise.
  • Hiding errors instead of returning them, leaving the model no way to recover.
  • Trusting the prompt to keep a tool safe instead of validating inputs in code.

🏁 Milestone: You have a model that correctly chooses among several tools, calls them with valid inputs, and recovers gracefully after a tool returns an error.

✎ Publish this: Write a short post titled something like "Your AI agent is not dumb, your tools are." Show one badly designed tool and the improved version side by side, and describe the behavior change you measured.
