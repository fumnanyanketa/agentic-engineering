🎯 Objective: Build an orchestrator-worker system and develop the judgment to know when not to.

The case FOR multi-agent (Anthropic): a multi-agent research system (Opus 4 lead + Sonnet 4 subagents) outran single-agent Opus 4 by 90.2% on their internal research eval. Multi-agent "excels for breadth-first queries that pursue multiple independent directions simultaneously." (multi-agent research system)

The case AGAINST (Cognition, "Don't Build Multi-Agents"): parallel multi-agent architectures are "very fragile" because subagents act on "conflicting assumptions not prescribed upfront." Their principles: (1) share full context and agent traces, not just messages; (2) actions carry implicit decisions, and conflicting decisions carry bad results. Their default: "just use a single-threaded linear agent," and when context overflows, add a dedicated LLM to compress history. (Cognition)

The reconciliation (both camps actually agree on the boundary): the split is read-heavy/parallelizable vs. write-heavy/dependent work. Anthropic itself concedes that "domains that require all agents to share the same context or involve many dependencies between agents are not a good fit" — which is precisely Cognition's thesis. So: - Multi-agent wins: research, breadth-first search, gathering — independent subtasks, info exceeding one context window. - Single-agent wins: sequential dependent work (e.g., editing a codebase) where "each action changes the state and later steps depend on exact earlier decisions."

The cost reality: agents use ~4× more tokens than chat; multi-agent ~15×. Multi-agent only pays off "where the value of the task is high enough." The 90.2% figure is an internal eval, not an independent benchmark — pair it mentally with that 15× cost. (Anthropic · LangChain: how/when to build multi-agent)

Claude layer: Use the Agent SDK's subagents to implement an orchestrator-worker pattern: a lead agent decomposes, spawns workers in isolated contexts, each returns a distilled summary, the lead synthesizes.

Lab 10: Build an orchestrator-worker research system: a lead agent that decomposes a research question, dispatches N worker agents (each in its own context, each using MCP tools/retrieval), and synthesizes their distilled summaries into one answer. Then build the single-agent version of the same task and compare quality, latency, and token cost. Write down which you'd ship and why.

✅ Best practices: reach for multi-agent only for breadth-first/parallelizable/high-value tasks; share full traces to workers; always compare against the single-agent baseline on cost, not just quality.

⛔ Pitfalls: multi-agent as a default ("it sounds sophisticated"); ignoring the 15× cost; parallelizing dependent work and getting incoherent merges.

🏁 Milestone: A working orchestrator-worker system and a reasoned, numbers-backed decision about whether it beats the single-agent baseline for your task.

✎ Publish this: "Should you build multi-agent systems? Anthropic says +90%. Cognition says don't. Here's the line between them." (Excellent flagship piece — genuine expert disagreement, no hype.)
