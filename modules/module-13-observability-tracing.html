🎯 Objective: Learn to see inside your agents while they run in production. By the end you can watch every step, every token, every dollar, and every delay, and replay any run that goes wrong instead of guessing what happened.

First, one word to define. Observability means how easily you can understand what a running system is doing from the outside, just by looking at the signals it gives off. A car with a dashboard (speed, fuel, engine temperature) is observable. A car with no gauges is not. This module is about putting a good dashboard on your agents.

This matters more for agents than for ordinary software. A normal program runs the same steps every time. An agent built on an LLM (a Large Language Model, the kind of AI that predicts text one chunk at a time) makes different choices on different runs. It may call a tool, read the result, change its mind, and try again. When it misbehaves, you cannot just re-run it and expect the same path. You need a recording of what it actually did.

The three pillars: logs, metrics, and traces

People who run software in production talk about three kinds of signals. They are often called the "three pillars" of observability. Each answers a different question.

  • Logs are timestamped text records of events, like a diary the system writes as it goes ("called the search tool", "got 5 results"). They answer "what happened, in words?" The classic advice is to log generously, because a log you did not write is a clue you will never get back.
  • Metrics are numbers measured over time, such as how many requests per minute or how long responses take. They answer "how much, and how fast?" Three metrics matter a lot for LLM systems. Time to first token (often written TTFT) is how long the user waits before the very first chunk of the answer appears. Tokens per second is how quickly the rest of the answer streams out (a token is a small chunk of text, usually part of a word). Total latency is the full wait from request to finished answer. ("Latency" just means delay.)
  • Traces are the full step-by-step path of a single request, from the moment it arrives to the moment it finishes, with timing and cost recorded for each step. They answer "what was the exact sequence, and where did the time and money go?" Traces are the pillar that makes agents debuggable, so most of this module is about them.

The key habit is to add all three from the very beginning, not after your first outage. Chip Huyen's widely shared guide to building a generative-AI platform makes the same point: observability is part of the platform, not an afterthought bolted on once something is already on fire (Huyen, Building a GenAI platform).

Spans and traces: how to see inside a multi-step agent

To debug an agent you need to break one run into pieces. The unit is called a span. A span is one timed operation: a single model call, a single tool call, or one sub-agent doing its job. Each span records when it started, when it ended, and useful details (which tool, how many tokens, what it cost).

A trace is the whole tree of spans for one request. The top span is the user's request. Underneath it sit the model calls and tool calls it triggered, and underneath those sit any further calls those steps caused. Reading a trace is like reading a receipt with sub-totals: you can see exactly which step was slow, which step was expensive, and which step returned the wrong thing.

Here is a small example. Imagine a user asks an agent, "What did we spend on cloud hosting last month?" A trace of that one request might look like this:

  • Span 1 (root): handle user question. Total time 6.2 seconds, total cost a few cents.
  • Span 2: first model call, decides to use a database tool. 1.1 seconds, 900 input tokens, 40 output tokens.
  • Span 3: tool call to the billing database. 0.3 seconds, returns rows.
  • Span 4: second model call, writes the final answer. 4.8 seconds, 4,200 input tokens.

One glance tells you the slow, costly step is Span 4, and that it swallowed 4,200 input tokens. That is your clue: something is stuffing too much text into the final model call. Without the trace, you would only know "the answer took 6 seconds" and have nowhere to start.

OpenTelemetry: a shared language so you are not locked in

OpenTelemetry (usually shortened to OTel) is an open, vendor-neutral standard for producing logs, metrics, and traces. It is hosted by the Cloud Native Computing Foundation (the CNCF, a neutral non-profit that stewards widely used infrastructure projects). The idea is simple: you describe your spans using one agreed set of names, and then any tool that understands those names can read your data. You instrument once and stay portable.

OTel has a special set of rules for AI systems called the GenAI semantic conventions. "Semantic conventions" just means an agreed naming scheme: agreed field names for the model used, the input and output token counts, the cost, the tool and agent calls, and why the model stopped. For example, the standard records the model name under gen_ai.request.model and token usage under gen_ai.usage.input_tokens and gen_ai.usage.output_tokens. Because the names are fixed, a span produced by one framework looks the same as a span produced by a hand-written call to a different model provider. Major platforms (including Google Cloud, AWS, Azure, and Datadog) now read these conventions, so your instrumentation is not tied to any one vendor (OpenTelemetry, GenAI observability · MLflow, OTel GenAI semantic conventions).

Tracing platforms: tools, not a backbone

You do not have to build a trace viewer yourself. Several tools store traces and draw them as readable trees. Treat them as interchangeable options, and lean toward ones that speak OpenTelemetry so you can switch later without rewriting your instrumentation. Some neutral examples, in no particular order:

  • Arize Phoenix, an open-source tracing tool built around OpenTelemetry.
  • Langfuse, an open-source option you can run on your own servers (self-host).
  • LangSmith, which is convenient if you already use the LangChain framework.
  • Braintrust, which ties tracing closely to running evaluations in your release pipeline.
  • General-purpose platforms such as Datadog that now read the OTel GenAI conventions directly.

A practical note on the "which is best" question: most published comparisons are written by the vendors themselves, so read them as marketing, not as a referee's verdict. The durable advice is vendor-neutral. Pick a tool that emits OpenTelemetry data, instrument once, and keep the freedom to move.

Lab 13: Add full tracing to the multi-step agent you built in Phase 4. Pick any OpenTelemetry-friendly tracing tool (an open-source one like Arize Phoenix or Langfuse is an easy free start). Then do three things. (1) Run one genuinely complex request and capture a trace view that shows every span, its latency, and its token cost. (2) Read the trace and find one inefficiency, for example a redundant model call, a tool you call twice, or a step bloated with too many input tokens. (3) Fix that one thing, re-run, and capture the new trace to prove the step got faster or cheaper. Save both traces side by side.

✅ Best practices:

  • Instrument from day one, before anything goes wrong, not after.
  • Use OpenTelemetry GenAI conventions so your data stays portable across tools.
  • Trace cost and latency, not just whether the answer was correct. A right answer that costs too much or takes too long is still a problem.
  • Record token counts on every model span, because they feed the cost work in Module 15.

⛔ Pitfalls:

  • Adding observability only after an outage, when the run you needed is already gone.
  • Keeping logs but no traces, so you cannot reconstruct a multi-step failure (logs tell you events happened, but not the tree that connects them).
  • Locking yourself in with one vendor's proprietary instrumentation that no other tool can read.
  • Measuring only correctness and being blindsided later by the bill or the wait times.

🏁 Milestone: You can produce a fully traced run of your agent that shows per-step cost and latency, point to the slowest or most expensive span, and show one trace-driven optimization with before-and-after numbers.

✎ Publish this: Write a short walkthrough titled "You can't fix what you can't see: tracing an AI agent in production." Show one real trace, name the inefficiency you found in it, and explain the fix and the savings.
