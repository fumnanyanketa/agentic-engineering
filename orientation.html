How to read this playbook

This page is your map. The course itself is a sequence of modules that take you from "I have never called a model" to "I can build, test, and run an agent system other people depend on." Here is how the pieces fit together.

  1. A hands-on learning path. Work through it module by module. Each module has clear objectives, the core concepts in plain language, a hands-on lab (a small thing you actually build), best practices, common pitfalls, an assessment milestone, and a short reading list. The labs add up. By the end they form a capstone, which is one larger project that combines everything: a small system where several agents work together to finish a task.
  2. A path you can share and learn out loud. The writing is plain enough that any motivated beginner can follow it. Each module ends with a "Publish this" prompt: a short writing or demo idea you can post the same week you learn the topic. Teaching a thing back, in your own words, is one of the fastest ways to make it stick. You do not have to publish anything to finish the course; the prompt is just an option.

Two layers, in every module. Each topic is taught as a vendor-neutral concept first (the idea that stays true no matter which company's model or tools you use), and then as a concrete way to try it yourself using whatever LLM provider account and tools you have. An LLM is a Large Language Model, the kind of AI that powers chat assistants. An LLM provider is any company that lets you send text to one of their models and get a response back, usually through an API (Application Programming Interface, a way for your code to talk to their service). The concepts are the durable part you keep for years. The specific tool you happen to use is just how you get your hands dirty today.

A note on honesty. This field is loud and full of hype. Where something is marketing dressed up as insight, this course says so plainly. Where credible experts genuinely disagree (and they do, sharply, on topics like multi-agent systems and how to test AI), you will get both sides and a recommendation, not a fake tidy consensus.

What "agentic engineering" actually is (and is not)

Start here so the rest of the course has a frame.

AI engineering is the practice of building applications on top of existing models, rather than training new models from scratch. A foundation model is a large, general-purpose model (such as an LLM) that has already been trained by someone else, so you can use it right away. The term "AI engineer" was popularized by Shawn "swyx" Wang's 2023 essay "The Rise of the AI Engineer." Its point was that foundation models collapsed a range of AI tasks that "used to take five years and a research team" into something you can now do "with API docs and a spare afternoon." Chip Huyen, in her book AI Engineering (O'Reilly, 2025), puts it bluntly: it is "just software engineering with AI models thrown in the stack," and it centers on adapting and evaluating models, not building them. (swyx · Pragmatic Engineer on the AI engineering stack)

Agentic engineering is the part of AI engineering that focuses on the more autonomous end: systems where the model decides what to do next, calls tools, keeps track of state (what has happened so far), and recovers when something fails. An agent, in the plainest definition, is an LLM that runs tools in a loop to reach a goal. The more the model is allowed to direct its own steps, the more "agentic" the system is. And the more agentic it is, the more your job shifts from writing every step yourself to setting up the conditions under which the model produces good steps on its own. (Latent Space, "Agent Engineering")

How it differs from nearby roles:

Role Starts from Core activity Skips
ML Engineer Data Trains and deploys models from scratch, and prepares the data they learn from (nothing skipped; this is the from-scratch role)
AI / Agentic Engineer A working foundation model Adapts, orchestrates, evaluates, and operates models in real products Training from scratch, and the heavy research math
Prompt Engineer A prompt Crafts the instructions that steer one model's behavior The systems and operations work around the model

The shift that defines the discipline: traditional machine learning begins with collecting data and training a model, while AI engineering enables a product-first approach, where you build on an existing model first and adapt only if you need to. (ML here is short for machine learning, the broader field of teaching computers from data.) (Notes on AI Engineering, Chapter 1)

Two honest caveats, up front:

  • The claim that "ML knowledge is no longer a must-have for building AI applications" is a direction, not a settled fact. Even people who say it admit some ML knowledge helps when you are troubleshooting. You need enough understanding to reason about why a model failed. You do not need to derive the underlying math. Module 1 draws that line on purpose.
  • "Prompt engineering" is being folded into the broader role, not crowned as a standalone career. A study of real job postings found that prompt-engineer-only roles are well under 1% of listings. Treat prompting as a skill inside agentic engineering, not a destination. (job-market study)

What you need before you start (prerequisites)

You do not need a machine learning background. You do not need advanced math. You do need a few practical things in place so the labs go smoothly:

  • Comfort writing small programs. If you can read and write basic code in any one language (Python is the most common in this space, but the ideas carry over), you are ready. The labs are short scripts, not large applications.
  • A code editor and a terminal. A terminal is the text window where you type commands to your computer. You will run small scripts and install packages from it.
  • Basic version control with Git. Git is the standard tool for saving snapshots of your code and tracking changes. You only need the basics: save a change, see what changed, undo a mistake.
  • An LLM provider account with API access. Pick any reputable provider that gives you an API key (a secret token that lets your code call their model). Most providers offer low-cost, pay-as-you-go access that is more than enough for these labs. The course is written so it works with any provider; nothing here depends on one specific company.
  • A small budget for API usage. The labs send real requests, and most providers charge per token (a token is a small chunk of text). The total cost of working through the course is small, but it is not zero. Set a spending limit in your provider's dashboard so you never get a surprise bill.

If any of these are new to you, Module 0 covers the baseline skills and the right mental model before the technical content begins.

The shape of the journey

Six phases, about 17 weeks at roughly 10 focused hours per week. You can compress or stretch this freely. The order matters more than the calendar. Each phase ends with a small working result, and those results stack up into the final capstone.

PHASE 0  Orientation & prerequisites            Week 0      → dev environment + mental models
PHASE 1  Foundations: the single model          Weeks 1–3   → a reliable single-call feature
PHASE 2  The augmented LLM                       Weeks 4–6   → a tool-using, retrieval-grounded assistant
PHASE 3  Workflows & agents                      Weeks 7–9   → patterned workflows + your first real agent
PHASE 4  Orchestration & multi-agent             Weeks 10–11 → an orchestrator-worker system
PHASE 5  Reliability: evals, observability,      Weeks 12–14 → an evaluated, traced, hardened system
         safety, production
PHASE 6  Practice & capstone                     Weeks 15–17 → coding-agent craft → final capstone

The one principle that governs all six phases is the point that nearly every credible source agrees on, with almost no dissent: start with the simplest thing that could work, and add agentic complexity only when it clearly improves the result. As one widely cited engineering guide puts it, find the simplest solution possible and only increase complexity when needed. For many problems, a single well-crafted model call (optionally with some retrieved information and a few examples) is already enough. Internalize this now. It will save you from most of the mistakes people make in this field. (See the reading shelf in Resources for the primary sources behind this principle.)

How you know you are learning (assessment)

Two simple checks run through the whole course:

  1. The capstone, built a piece at a time. Every lab is one component of the final system. If your labs work and fit together, you are learning. If they do not, you have found exactly what to revisit.
  2. An eval portfolio. An eval (short for evaluation) is a test for an AI system: a set of example inputs plus a way to grade whether the output was good. Starting in Phase 2, you keep a growing collection of evals for your own work. By the end you should be able to say, with actual numbers, how reliable each part of your system is. In this field, saying "it works" without an eval is just a guess.

The exit bar (your real proof of skill): ship a small agent system that (a) breaks a non-trivial task into steps handled by specialized workers, (b) uses external tools, ideally through MCP (the Model Context Protocol, an open standard for connecting models to tools and data), (c) manages its context across long tasks, (d) has an eval suite with reported pass rates, (e) can be traced and observed (you can see what it did and why), (f) defends against the main security risks for agents, and (g) has documented cost and time-per-task. That is a portfolio piece you can show to anyone.
