What "agentic engineering" actually is (and isn't)

Start here so the rest has a frame.

AI engineering is the discipline of building applications on top of foundation models rather than training models from scratch. The term was popularized by Shawn "swyx" Wang's 2023 essay "The Rise of the AI Engineer," whose thesis was that foundation models collapsed "a wide range of AI tasks that used to take five years and a research team" into something achievable "with API docs and a spare afternoon." Chip Huyen's framing in AI Engineering (O'Reilly, 2025) is blunter: it's "just software engineering with AI models thrown in the stack," centered on model adaptation and evaluation, not model building. (swyx · Pragmatic Engineer on Huyen's stack)

Agentic engineering is the sub-discipline focused on the autonomous end of that spectrum: systems "where a language model decides what to do next, calls tools, holds state, and recovers from failure." The more an LLM directs its own control flow, the more "agentic" the system — and the more your job shifts from writing the logic to engineering the conditions under which the model produces good logic. (Latent Space, "Agent Engineering")

How it differs from neighbors:

Role Starts from Core activity Skips
ML Engineer Data Train/deploy models from scratch; feature engineering —
AI / Agentic Engineer A working foundation model Adapt, orchestrate, evaluate, operate Training from scratch; research math
Prompt Engineer A prompt Craft instructions controlling one model's behavior Systems/ops

The inversion that defines the discipline: traditional ML "begins with data collection and model training," while AI engineering enables a product-first approach — build on an existing model first, adapt later. (Notes on Huyen Ch.1)

Two honest caveats, up front:

  • Huyen's provocation that "ML knowledge is no longer a must-have for building AI applications" is directional, not settled — even she hedges that it helps for troubleshooting. You need enough ML to reason about failures; you don't need to derive backprop. This curriculum's Module 1 draws that line deliberately.
  • "Prompt engineering" is being absorbed into the broader role, not crowned as a standalone career. An empirical job-postings study found prompt-engineer-only roles at well under 1% of listings. Treat prompting as a skill inside agentic engineering, not a destination. (job-market study)

The shape of the journey

Six phases, ~17 weeks at ~10 focused hours/week. Compress or stretch freely — the sequence matters more than the calendar. Each phase ends with a working artifact; the artifacts stack into the capstone.

PHASE 0  Orientation & prerequisites            Week 0      → dev environment + mental models
PHASE 1  Foundations: the single model          Weeks 1–3   → a reliable single-call feature
PHASE 2  The augmented LLM                       Weeks 4–6   → a tool-using, retrieval-grounded assistant
PHASE 3  Workflows & agents                      Weeks 7–9   → patterned workflows + your first real agent
PHASE 4  Orchestration & multi-agent             Weeks 10–11 → an orchestrator-worker system
PHASE 5  Reliability: evals, observability,      Weeks 12–14 → an evaluated, traced, hardened system
         safety, production
PHASE 6  Practice & capstone                     Weeks 15–17 → coding-agent craft → Atlas OS

The one principle that governs all six phases (every credible source agreed on this — it was the single point with zero dissent): start with the simplest thing that could work, and add agentic complexity only when it demonstrably improves outcomes. Anthropic: "find the simplest solution possible, and only increasing complexity when needed." For many problems, "optimizing single LLM calls with retrieval and in-context examples is usually enough." Internalize this now; it will save you from 80% of the field's mistakes. (Building Effective Agents)

Assessment model (how you know you're learning)

Two instruments run the whole way through:

  1. The capstone, built incrementally. Every lab is a component of the final orchestration system. If your labs work and compose, you're learning.
  2. An eval portfolio. From Phase 2 onward you keep a growing set of evals (test cases + graders) for your own work. By the end you should be able to say, with numbers, how reliable each component is. In this field, "it works" without an eval is just a vibe.

Capstone exit bar (the real diploma): ship an agent-orchestration system that (a) decomposes a non-trivial task across specialized workers, (b) uses tools via MCP, (c) manages context across long horizons, (d) has an eval suite with reported pass rates, (e) is traced/observable, (f) defends against the lethal trifecta, and (g) has documented cost/latency per task. Publishable as a build-in-public series.
