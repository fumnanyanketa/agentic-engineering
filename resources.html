Appendix A: The anti-hype reading shelf (primary sources first)

These are the main sources behind this course, with primary sources (the original work, not someone's summary of it) listed first. When in doubt, trust these over secondhand blog posts. A few entries below are written by specific companies, including model providers. Those are cited as useful sources, one among several, not as the platform this course is built on. Read company-authored pieces knowing the author has a product to sell, and cross-check claims against the neutral sources.

Foundations of the discipline: - Chip Huyen, AI Engineering (O'Reilly, 2025), the standard book on building applications on top of models. book · code repository - swyx, "The Rise of the AI Engineer," the essay that named the role. ai.engineer - Andrej Karpathy, "Neural Networks: Zero to Hero," a free video course that builds up the basics from scratch. karpathy.ai

How to build agents (start with the vendor-neutral idea, then compare guides): - "Building Effective Agents," a widely cited engineering guide from a model provider, argues for simple patterns first and adding complexity only when it pays off. read it here - "A Practical Guide to Building Agents," a free guide from another model provider, covers when to build an agent, design foundations, and guardrails. Reading two providers side by side is a good way to separate shared principles from house style. PDF - "Effective context engineering for AI agents," on how to manage what goes into the model's limited context window. read it here - "How we built our multi-agent research system," a worked example of several agents cooperating (note the cost trade-offs it reports). read it here - "Writing tools for agents," practical advice on designing the tools an agent calls. read it here - "Demystifying evals for AI agents," an introduction to testing agent behavior. read it here

The counterpoints (read the disagreements on purpose): - Cognition, "Don't Build Multi-Agents," argues that many tasks are better served by a single agent. cognition.ai - LangChain, "How and when to build multi-agent systems," takes a middle position. langchain.com

These two disagree with the multi-agent enthusiasm above. That tension is real and unsettled. Reading both sides is the point.

Core research papers (the ideas many tools are built on): - ReAct (Yao et al., 2022), which combines reasoning and tool use in a loop. arXiv:2210.03629 - Reflexion (Shinn et al., 2023), where an agent reflects on its own mistakes to do better next time. arXiv:2303.11366

Evaluation and safety: - Hamel Husain and Shreya Shankar, evals FAQ, a practical question-and-answer guide to testing LLM systems. hamel.dev - Eugene Yan, "Evaluating LLM-Evaluators," on the common practice of using one model to grade another, and its pitfalls. eugeneyan.com - Chip Huyen, "Building a Generative AI Platform," a vendor-neutral tour of the moving parts in a real system. huyenchip.com - Simon Willison, "The lethal trifecta," explains the security risk when an agent has access to private data, exposure to untrusted content, and the ability to send data out, all at once. simonwillison.net

Working with coding agents (the backbone of Module 16, and a thread through the course): - "Agentic Engineering Patterns," an evolving free guide to getting good results from coding agents (how they work, version control, subagents, test-driven development, manual testing, annotated prompts). Re-check it for new chapters, since they are added over time. simonwillison.net/guides - "Designing agentic loops," on the core skill of building loops with clear success criteria and self-checking, run inside a sandbox with spending caps. simonwillison.net - "I think 'agent' may finally have a widely agreed definition," which lands on a simple one: an agent runs tools in a loop to reach a goal. simonwillison.net - "Here's how I use LLMs to help me write code," on managing context, the testing you cannot hand off, and why a model is "fancy autocomplete" rather than a colleague. simonwillison.net - "Not all AI-assisted programming is vibe coding," which draws the precise line between casually generating code and actually engineering software. simonwillison.net - "Embracing the parallel coding agent lifestyle," on review becoming the bottleneck, and running several research or scouting tasks in parallel. simonwillison.net - "Prompt injection" (the original 2022 post), where the term came from and why it is hard to detect. simonwillison.net

Standards and official documentation: - MCP (Model Context Protocol) is an open standard for connecting models to tools and data. Start with the official introduction. modelcontextprotocol.io - Your LLM provider's own documentation is the source of truth for current model names, context-window sizes, pricing, tool-use formats, and features like prompt caching (reusing repeated context to cut cost). Provider docs change often, so always check the current version rather than trusting a number you read somewhere. Treat any specific guide, including provider docs, as one input rather than the last word.

Appendix B: Quick-reference decision rules

These are short rules of thumb you will reach for again and again. Each one names a common choice and gives a simple default. A few terms used below: RAG (Retrieval-Augmented Generation) means fetching relevant text and giving it to the model so its answer is grounded in your data; fine-tuning means further training a model on your own examples to shift its style or behavior; a token is the small chunk of text that providers count for cost.

  • Workflow or agent? If the steps are predictable, use a fixed workflow (you decide the steps in advance). If the steps or the length of the task are unpredictable, use an agent (the model decides the next step). Default to the simplest option that works.
  • Single agent or multiple agents? For tasks that are mostly reading, can be split into independent parts, and are worth the extra cost, multiple agents can help. For tasks that are mostly writing, where each step depends on the last, a single agent is usually better. Always compare cost, not just quality: running many agents can use many times more tokens than one.
  • Prompt, RAG, or fine-tune? Try a plain prompt first. Add RAG when the task needs your own data or fresh, up-to-date information. Reach for fine-tuning only for matching a specific style or voice, or for a narrow high-volume task, or when prompting plus RAG still miss your quality bar.
  • Which model size? Pick the cheapest, smallest model that still passes your eval. After any change you make to save money, run the eval again to confirm quality did not drop.
  • Framework or raw API? Learn the raw API first so you understand the underlying loop. Adopt a framework (a library that handles boilerplate for you) once you know what it is doing for you, chosen to fit the shape of your problem.
  • Vibe coding or agentic engineering? If you are shipping agent-written code you have not reviewed, that is "vibe coding." Agentic engineering means a verification loop (tests and type checks), reviewing every change before it lands, and running the agent in a sandbox (an isolated environment where mistakes cannot do harm). You stay accountable for the output.
  • Is it safe? Check for the lethal trifecta: access to private data, exposure to untrusted content, and a way to send data out. If all three are present at once, redesign to remove one of them. Require a human to approve any consequential action. Sandbox any agent that runs code.
  • Is it done? Not until it has an eval with a reported pass rate. Saying "it works" without an eval is a guess, not an engineering claim.

Appendix C: Honest caveats about this course

  • The field moves fast. Specific tooling (frameworks, vector databases for storing text for retrieval, model names and prices, even how MCP is governed) will drift over time. The concepts in this course are durable. Always re-verify the specific tools and any provider details against current documentation before you rely on them.
  • Some sources are company marketing. Rankings of eval platforms and frameworks often come from competing companies' own blogs. They are directionally useful, not neutral benchmarks. The primary sources in Appendix A carry the real weight, and even those should be read with the author's interests in mind.
  • Genuine disagreements are left intact, on purpose. Whether to use multiple agents, and how best to evaluate systems, are open questions where credible experts disagree. A course that pretended otherwise would be misleading you. Form your own view by building and measuring.
  • Numbers are illustrative unless they come from a primary source. Figures you see quoted (token-cost multipliers, context-window limits, headline accuracy gains) are directional signals. Check the original context before you repeat them.
